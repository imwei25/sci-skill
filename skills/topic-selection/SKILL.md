# 选题 / 假设生成技能

把"想做研究"变成几个**具体、可评估、可执行**的候选课题。参考 K-Dense 的 hypothesis-generation 思路。

## 范围自检（开工第一件事）
本套件**主控优先：默认所有请求先经 `sci-pilot` 统一调度**。
- **先看有没有被派发**：上文若出现派发标记 `[sci-pilot派发·…·直接执行]`、或已有 workspace 上下文、或你是被 sci-pilot/其它技能作为其流程的一步调用 → **直接做你这一步，别往回绕**（防兜圈）。
- **否则**（用户冷启动直接触发、上文无任何派发痕迹）→ **先交回 `sci-pilot`** 判意图与范围；它判为单步会带派发标记把这一步立刻派回来、你再就地做（对用户只多“判一下意图”这一步）。

## Python 环境（可选）
> 没有项目根 `.venv`？先运行 `env-setup` 技能建好并装依赖。
```
.venv/Scripts/python.exe   # Windows（正斜杠写法，bash 与 PowerShell 都能用）
.venv/bin/python           # Linux / macOS
```

## 流程
1. **意图对齐（先写 brief，别急着查）**：主代理先把理解到的用户意图复述成一份**研究简报（research brief）**，写清：目标方向、范围与边界、隐含约束（学科/子领域、手上已有的数据或样本、可用方法与周期/经费、新颖性门槛、目标期刊或资助）。缺的约束**主动问用户**（如"你手上有什么数据/样本？""想冲顶刊还是求稳发出去？""偏机制研究还是临床应用？"）。若是高投入调研，**先把 brief 给用户确认一次**再往下。这份 brief 是后续所有子任务共读的同一份规格。
2. **摸清现状**：按 brief 用 `research-scan` 或 `literature-review` 拿真实文献，梳理该方向的空白与争议（别拍脑袋）。子问题多且相互独立时可并行取证，否则串行；**运行时不支持子代理就串行**。
3. **生成候选课题**：围绕空白提 3-6 个，每个写清：
   - **一句话课题名**
   - **科学假设**（可证伪的陈述）
   - **PICO**（临床题）：Population / Intervention / Comparison / Outcome
   - **拟用方法/数据**
   - **预期贡献**
4. **多棱镜评分**：对五个维度各评 **1–5 分**（每个分数都要附一句证据/理由，别只给数字），汇总排序——**支持子代理时，每维派一个专门 lens 的评审各自独立取证打分**，主代理综合；不支持时由主代理逐维度顺序自评。这样避免单一视角的乐观偏差。
   | 维度 | 说明 | 打分方向 |
   |---|---|---|
   | 新颖性 | 是否已有人做透；增量还是突破 | 越新越高 |
   | 可行性 | 数据/样本/技术/周期/经费是否够 | 越可行越高 |
   | 价值 | 临床或科学意义大小 | 越大越高 |
   | 竞争度 | 热门=易撞车，冷门=易被忽视 | **适中最高**（太热撞车、太冷无人问津都扣分） |
   | 伦理/合规可行性 | 涉及人/动物/人遗资源/隐私的审批与合规难度 | 越可行越高；**根本过不了伦理→一票否决** |
   - **与申请人条件匹配**也要纳入考量（brief 里问过的手上数据/样本/方法/周期）——再好的题，手上条件够不着也要下调可行性。
5. **引用核验（分两步，别把两件事混为一谈）**：
   - **存在性核验**：把候选与评分所引文献先导出成 `outputs/topic_refs.txt`（每行一条 DOI/PMID/标题），交 `reference-check` 技能核验**真实存在、标题吻合**。⚠️ `reference-check` 只查"这篇文献是否真的存在"，**不能判断它是否支持你的论断**。
   - **支持性核验**：文献"是否真的支持该论断"由主代理**自己回读摘要**确认（必要时用 `fulltext-retrieval` 下全文再读）。别因为脚本跑完就宣称"已核实支持"——那恰恰是本步要防的编造。
6. **完备性自查**：收尾前自问"还漏了什么——没查的来源、没验证的空白判断、没考虑的相邻方向？"，有明显缺口就补一轮。若因成本只查了部分来源/方向，**如实说明**，别让用户误以为覆盖完整。
7. **推荐 + 下一步**：推荐 1-2 个主攻 + 1 个备选，说明理由与主要风险。然后**主动询问用户**要不要就他最感兴趣的方向做一次**对抗式研究**（见下），并解释清楚它是什么、这次做它的意义——**用户同意才跑**（成本较高，不自动执行）。

## 对抗式研究（按需，需用户确认）
- **是什么**：不再找支持该方向的证据，而是**反过来尽力推翻它**——派一个"反方"子代理专门去找：这个 idea 是否已被人做过（及做到什么程度）、有没有致命的不可行点、隐含假设是否站得住。默认"存疑即真"，能扛住反驳的方向才算真新颖、真可行。
- **意义**：选题阶段最贵的错误，是投入大量精力后才发现方向早被做过或根本走不通。对抗式研究用一次集中的证伪把这个风险提前暴露；同时顺带产出**邻近工作地图**——别人在相似方向做了什么、你的增量差异在哪。
- **为何按需**：它要额外多轮检索与子代理，成本明显高于常规选题，所以只对用户**最感兴趣的 1 个方向**、在用户同意后做。
- 需要时直接衔接 `deep-research`（做该方向的 prior-art / 新颖性复核）与 `research-scan`（画邻近工作地图）。

## 约定
- 产出写 `outputs/topics.md`（含 brief 摘要、候选表、评分、推荐、风险、下一步建议）。
- **出 PDF（正式报告默认交付版）**：写完后把 `outputs/topics.md` 交给 `render-pdf-doc` 技能渲染成 `outputs/topics.pdf`。中文报告务必指定中文字体（`--cjk-font`：本地 `Microsoft YaHei`，服务器 `Noto Sans CJK SC`），否则会漏字。
- 假设必须可证伪、边界清楚，别写成宽泛口号。
- 空白判断要有文献支撑；**不要虚构"尚无人研究"**——拿不准就说需进一步检索确认。
- 区分**检索事实 / 模型推断 / 观点**；关键论断至少 2 个独立来源，冲突就并列标分歧。
- 衔接：选定 1–2 个主攻方向后，动手写标书/建库前先用 `novelty-check` 做**严格新颖性裁定 + 预注册锁**（把这里的"新颖性打分"升级为可辩护的裁定，并在采数前冻结假设与分析计划）；之后 `grant-proposal` 写标书、`literature-review` 做综述；深挖用 `deep-research`。
